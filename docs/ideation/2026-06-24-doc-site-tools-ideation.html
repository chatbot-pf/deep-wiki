Documentation tools beyond VitePress

Strong directions for letting the wiki packaging layer target generators other than VitePress — Docus, Fumadocs, and others — without forking the build for each one.

Date 2026-06-24 Topic doc-site-tools Focus add Docus / Fumadocs (and others) to wiki packaging Mode repo-grounded

Grounding Context

Where VitePress is baked in

  • skills/wiki-vitepress/SKILL.md (overview) + references/vitepress-build.md — ~650 lines of VitePress-specific config.mts, theme/index.ts, custom.css, package.json, and post-processing.
  • commands/build.md — thin entrypoint (disable-model-invocation: true) that delegates to the skill.
  • commands/deploy.md — generates a GitHub Actions workflow hardcoded to VitePress: npm run build, artifact path wiki/.vitepress/dist, VitePress base-path logic, Node 20.

Signature cross-cutting features (the expensive porting surface)

  • Dark-mode Mermaid "three-layer fix" (theme variables + CSS !important overrides + inline-style post-processing).
  • Click-to-zoom: a custom fullscreen SVG overlay — Mermaid renders <svg> not <img>, so it polls async SVGs, clones, fixes viewBox, pans/zooms, and cleans up listeners. The most bug-prone code in the skill.
  • Focus mode toggle, source-linked citations, canonical dark palette + Inter/JetBrains Mono, dynamic sidebar from numbered section folders, onboarding-first, llms.txt/AGENTS.md emission.
  • Markdown post-processing: <br/>→<br>, wrap bare <T> generics in backticks, validate Mermaid hex, ensure frontmatter.

External context — the 2026 tool landscape

  • Docus (Nuxt 4 / Vue, Nuxt Content / MDC): no built-in Mermaid (SSR-sensitive; needs <ClientOnly> or custom remark); built-in dark mode; static export nuxt build --preset github_pages → .output/public, base via NUXT_APP_BASE_URL; MDC --- delimiter can misparse plain markdown. Highest porting friction.
  • Fumadocs (Next.js / React; MDX or .md via local-md): Mermaid documented but not built-in (~50-line client component; clean dark via useTheme()); export next build (output:export) → out/ + .nojekyll; Node 22+; MDX chokes on bare <T> generics — use .md mode. Moderate effort.
  • Other strong contenders: Nextra v4 — built-in Mermaid (@theguild/remark-mermaid); Docusaurus — build-time Mermaid pre-render → dual light/dark SVG, zero JS, instant switch, largest community; Starlight (Astro) — lightest but Mermaid is its #2 missing feature; rspress, Mintlify (hosted) round out the field.
  • Burden ranking for deep-wiki's needs: Nextra v4 < Fumadocs < Docus. Key reframe: build-time Mermaid (Docusaurus, Nextra) natively does what VitePress needed ~300 lines of custom code for — and gives instant dark/light for free.

Topic Axes

  • A1Tool abstraction & skill architecture — separate skills per tool vs one parametrized skill vs shared-core + per-tool adapters.
  • A2Cross-cutting feature parity — porting dark Mermaid, SVG click-zoom, focus mode, and citations per tool.
  • A3Content / Markdown portability — markdown-it vs MDX (JSX / <T> breakage) vs MDC; the post-processing layer.
  • A4Tool selection & invocation UX — how build.md / deploy.md pick and stay aware of the chosen tool.
  • A5Maintenance, parity testing & contribution model — keeping N tools in sync, smoke-testing builds, version pinning.

1.

Shared core skill + declarative adapter contract

Axis
A1 · Tool abstraction & skill architecture
Basis
direct: references/vitepress-build.md is ~650 monolithic, tool-coupled lines, yet SKILL.md already separates a tool-agnostic overview from the authoritative reference; deploy.md hardcodes npm run build + wiki/.vitepress/dist. The palette table, post-processing rules, citations, sidebar, and "developer-focused landing page" are content concerns independent of any tool. external: the per-tool facts a manifest must encode (Docus → .output/public/NUXT_APP_BASE_URL; Fumadocs → out/+.nojekyll+Node 22).
Confidence
90%
Complexity
High

Description

Split wiki-vitepress into a generator-neutral core (e.g. wiki-site-core) holding the genuinely portable logic — Markdown post-processing, source-linked citations, the canonical dark palette and fonts as design tokens, dynamic sidebar + onboarding-first, llms.txt/AGENTS.md emission, the landing page — plus a thin per-tool adapter. Each adapter is a small declarative capability manifest: build_cmd, install_cmd, output_dir, mermaid_strategy, dark_mode, node_version, base_path_mechanism, parser_profile, and extra_files (e.g. .nojekyll) — read identically by build.md, deploy.md, and CI. VitePress is refactored to be the reference adapter, not a privileged substrate, which proves the contract isn't VitePress-shaped. The same contract doubles as the community contribution unit.

Rationale

This is the keystone every other idea rests on. It converts "support N tools" from N copies of a 650-line reference into 1 engine + N small descriptors, so each cross-cutting bugfix or palette change lands once and propagates to every site. Making VitePress implement the contract rather than be exempt from it is the discipline that stops the abstraction from leaking VitePress assumptions.

Downsides

Large up-front refactor with no immediate user-visible payoff; the manifest schema must be designed and versioned; if VitePress isn't honestly re-expressed as an adapter, the "neutral" core will quietly encode its quirks. Best sequenced as the first move precisely because everything else plugs into it.

2.

Capability-gated Mermaid strategy slot

Axis
A2 · Cross-cutting feature parity
Basis
direct: SKILL.md "Known Gotchas" name this the worst part — "Mermaid renders async — SVGs don't exist when onMounted fires. Must poll" and "Node text in Mermaid uses inline style with highest specificity — CSS alone won't fix it." external: Docusaurus pre-renders Mermaid to dual light/dark SVG with zero JS and instant switch; Nextra v4 ships built-in Mermaid — both natively do what VitePress needed ~300 lines for.
Confidence
85%
Complexity
Medium

Description

Treat "how a tool renders dark Mermaid" as one named strategy slot with three reusable implementations the adapter chooses between: native (Nextra's built-in remark plugin — zero work), build-time (Docusaurus-style dual light/dark SVG pre-render, e.g. via @mermaid-js/mermaid-cli), and runtime-overlay (VitePress's existing async-poll + clone + viewBox + pan/zoom + dark-CSS code, generalized). Make build-time the default contract for new tools and demote runtime-overlay to a VitePress legacy fallback. The packager emits the heavy ~300-line client stack only when the chosen adapter declares runtime-overlay.

Rationale

The single most bug-prone subsystem in the whole product is the one several target tools make unnecessary. This inverts the intuition that VitePress is "easiest": on a build-time tool the feature ships with less code, not more, and the fragile async-timing/listener-leak surface leaves maintenance entirely for those tools. A tool that later gains native Mermaid switches strategy in one line.

Downsides

Build-time pre-render needs dual SVGs emitted to keep instant theme switching (more build artifacts); click-to-zoom is a separate concern that may still want a small vanilla overlay even over pre-rendered SVG; two strategies (build-time + runtime-overlay) still must be maintained for the tools that need each.

3.

Target-aware Markdown normalizer

Axis
A3 · Content / Markdown portability
Basis
direct: current post-processing already "wraps bare <T> generic parameters in backticks outside code fences" and "replaces <br/> with <br> (Vue template compiler compatibility)" — both already flavor-specific. external: Fumadocs MDX chokes on bare <T> in prose (use .md mode or escape); Docus MDC's --- delimiter can misparse plain markdown (nuxt/content#1260).
Confidence
80%
Complexity
Medium

Description

Generalize today's VitePress-only post-processing into a flavor-aware normalizer keyed by the adapter's parser_profile. One canonical wiki document is generated tool-neutral; a single normalization seam then applies exactly the transforms the chosen target needs: markdown-it (VitePress — <br/>→<br>); mdx (Fumadocs/Nextra — escape bare <T> generics the JSX parser treats as elements, and prefer emitting .md over .mdx so local-md skips JSX parsing entirely); mdc (Docus — guard the --- props/slot delimiter from being read as a horizontal rule).

Rationale

Markdown portability is where silent breakage lives — a page with Map<string, T> and a stray --- must build cleanly on markdown-it, MDX, and MDC from identical source, without the wiki author knowing the difference. Centralizing transforms as per-profile lowering passes means a new tool declares its escaping rules once instead of editing a shared monolith, and "try it in Fumadocs" becomes a re-package rather than a re-run of the expensive generation pipeline.

Downsides

Requires enumerating each parser's failure modes accurately — a moving target as the tools evolve; emitting .md to dodge MDX limits forfeits MDX component power; some breakage classes (raw HTML passthrough) may need per-tool config beyond text transforms.

4.

Auto-detect + recommend tool selection

Axis
A4 · Tool selection & invocation UX
Basis
direct: build.md is a thin disable-model-invocation: true entrypoint passing $ARGUMENTS — the natural seam for a --tool flag; deploy.md Step 1 already keys prerequisites to wiki/package.json. external: the burden ranking (Nextra v4 < Fumadocs < Docus) and content-discriminating facts (Fumadocs MDX vs Mermaid-heavy, generic-laden content) give a principled default. reasoned: each framework marks its presence on disk (next.config, nuxt.config, .vitepress/), so detection is inference from existing signals.
Confidence
75%
Complexity
Medium

Description

Rather than make the user name a generator, /deep-wiki:build resolves the target once via arg > config > auto-detect > default. It sniffs repo signals (.vitepress/ → VitePress, nuxt.config + @nuxt/content → Docus, next.config + fumadocs-* → Fumadocs) and the wiki's own content (Mermaid-heavy + bare <T> generics → recommend a build-time-Mermaid tool over Fumadocs MDX), proposes a default by burden ranking with one line of reasoning, and accepts --tool=fumadocs to override. VitePress stays the zero-config default so existing users see no change, and deploy.md reads the same resolved target so the Actions workflow's build command, output dir, Node version, and .nojekyll follow automatically.

Rationale

Most users won't know which generator fits Mermaid-heavy, citation-dense wiki content; auto-detect plus content-aware recommendation turns a research decision into a zero-friction default with an escape hatch, and keeps /build a single command instead of spawning one command per tool. Matching the wiki's framework to the host repo's framework is the choice a user would make anyway.

Downsides

Detection heuristics can mis-fire (a repo carrying both Next and Nuxt deps); the content-aware recommendation needs the wiki already generated to inspect; a wrong silent default is worse than a prompt, so genuine ambiguity should trigger a confirmation rather than a guess.

5.

Golden-fixture parity harness

Axis
A5 · Maintenance, parity testing & contribution model
Basis
direct: SKILL.md "Known Gotchas" (async Mermaid, inline-style specificity, SSR document) enumerate exactly the failure modes worth asserting. reasoned: supporting 3–5 independently-versioned ecosystems means that without a smoke build, the first signal of breakage is a user's failed deploy; a shared fixture exercising the exact pain points is the minimum viable regression net.
Confidence
72%
Complexity
Medium

Description

Ship one canonical fixture wiki exercising every signature feature — a dark Mermaid diagram, a bare <T> generic, a source citation, an onboarding page, a stray ---, a <br/> — and build it through every adapter in a CI matrix, asserting invariants per tool: non-empty dist, Mermaid renders with the canonical dark palette, citations resolve, the normalizer survived, and the declared zoom/focus tier holds. An adapter isn't "supported" until green. Each adapter pins the upstream tool version it was validated against (Fumadocs Node 22+, VitePress 1.x / Node 20), so an upstream breaking release fails CI loudly instead of a user's deploy silently.

InvariantVitePressDocusaurusFumadocsDocus
Mermaid renders (dark)runtimebuild-timepluginClientOnly
<T> survives✓✓via .md✓
Citations resolve✓✓✓✓
Zoom / focus tierfullzoom onlydeclareddeclared

Rationale

The cost of multi-tool support isn't writing adapters — it's keeping them green as several upstreams move. A fixture + matrix converts "did we port everything?" and "does tool X still work?" from manual audits into red/green gates, and hands contributors a clear, automated bar to clear when adding a generator.

Downsides

CI cost and time scale with tool count; headless screenshot-diffing is itself flaky and maintenance-heavy (a DOM/asset-presence assertion may be a saner first cut than pixel diffing); version pinning invites staleness without a refresh cadence.

6.

No-build zero-dependency preview tier · wildcard

Axis
A1 · Tool abstraction & skill architecture
Basis
direct: the packaging layer assumes the output is a JS-framework SSG built with npm install + a bundler, yet the wiki content is just Markdown + Mermaid. reasoned: the heaviest, flakiest, slowest part of build/deploy is node_modules + bundler; for the common "I just want to read it" case that cost buys nothing, and a single HTML file widens deploy targets beyond CI-with-Node.
Confidence
60%
Complexity
Medium

Description

Challenge the buried assumption that wiki output must be a bundler-built JS-framework SSG. The wiki is just Markdown + Mermaid — so offer a zero-dependency tier: a self-contained index.html that loads markdown-it + Mermaid from a pinned CDN (or inlined), or a node-less SSG like mdBook / zola with no node_modules. This becomes the instant-preview / air-gapped / CI-cheap "read it now" default, with VitePress / Docus / Fumadocs as the polished "rich" tier.

Rationale

Reframes the problem one level up: much of the porting pain is incidental to choosing heavyweight SSGs at all. A no-build tier sidesteps it entirely for previews and constrained environments — and including a non-SSG target stress-tests whether the core abstraction (idea 1) is genuinely generator-neutral or secretly SSG-shaped.

Downsides

A single-file / CDN site loses built-in search, sidebar polish, and offline assets unless carefully inlined; a CDN dependency reintroduces a network requirement; it is a different output shape the parity harness must accommodate; and it risks scope creep beyond the user's "Docus / Fumadocs" framing — best treated as an opt-in tier, not the headline.
